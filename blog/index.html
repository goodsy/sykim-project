이 기술블로그에서 앞으로 다룰 내용

2025년 12월 12일 · 약 2분

이 블로그는 단순한 기술 정리 공간이 아니라,
실무에서의 고민과 선택, 그리고 그 결과를 기록하는 공간으로 운영할 예정이다.

백엔드 개발자로 일하며 느낀 점은,
기술 자체보다도 "어떤 상황에서 왜 이 선택을 했는지" 가 훨씬 중요하다는 것이다.

1. 정산·결제(PG) 시스템 실무 기록​

가장 오래 다뤄온 영역인 결제(PG) 정산 시스템을 중심으로,
다음과 같은 내용을 정리할 계획이다.

  • 정산 기준정보 설계 (수수료율, 계산 방식, 세금계산서 기준)
  • 정산 배치 시스템 설계와 고도화 과정
  • 정산 중복 처리와 데이터 정합성 문제
  • PG/외부 시스템 연동 시 고려해야 할 포인트

실제 운영 환경에서 겪었던 문제와
그 문제를 해결하기 위해 어떤 구조와 방식을 선택했는지를 기록한다.

2. Spring Batch 기반 대량 처리와 성능 개선​

대량 데이터 처리와 배치는 정산 시스템에서 빠질 수 없는 영역이다.

  • Chunk vs Tasklet 선택 기준
  • Skip / Retry 정책 설계 경험
  • 대량 Insert / Update 성능 개선
  • 배치 장애 발생 시 재처리 전략

단순 기능 설명이 아닌,
운영 안정성과 성능을 함께 고려한 설계 관점을 중심으로 정리할 예정이다.

3. 트랜잭션, 동시성, 데이터 정합성​

실무에서 가장 많은 고민이 필요한 주제 중 하나다.

  • 트랜잭션 Isolation Level 선택 기준
  • Deadlock 발생 원인과 대응 경험
  • 동시성 충돌을 막기 위한 설계 방식

이론보다는 실제 문제 상황과 선택의 결과를 중심으로 다룬다.

4. 운영, 장애 대응, 그리고 자동화​

서비스는 개발보다 운영에서 더 많은 배움을 준다.

  • 장애 알림 설계
  • 로그와 이력 기반 원인 분석
  • 재처리와 롤백 전략
  • 운영 효율을 높이기 위한 자동화

“장애를 줄이는 코드”보다
**"장애를 빨리 파악하고 복구할 수 있는 구조"**에 대한 고민을 담을 예정이다.

이 블로그의 글들은 완벽한 정답을 제시하기보다는,
실무에서 고민하고 선택했던 과정을 솔직하게 기록하는 데 목적이 있다.

앞으로 이 공간에 쌓일 기록들이
스스로를 돌아보는 자료이자,
필요한 누군가에게는 작은 참고가 되었으면 한다.

Batch 레거시 정산 시스템 고도화

2025년 12월 12일 · 약 1분

Batch 정산 시스템 개선 사항​

1. select - insert 구조 기반으로 기준정보와 거래원장 join으로 정산 기준정보 조회와 계산하는 로직으로 구현 됨

  • Job 또는 Step 실행 시, 기준정보 캐싱 조회 -> 추후 Redis 연동
  • 캐싱된 기준정보와 정산 대 거래 원장의 정산 정보 생성

2. Domain Layer, Batch Layer, Infra Layer 계층으로 재구조

  • 데이터 정합성관련 설계
    • Table Unique Key

3. Retry / Skip 정책

  • Retry : 외부 연동 실패 시
  • Skip
    • 정책 정의 필요
      • 기준정보 누락
      • 거래금액 오류
    • 각 정책의 Skip 건수가 일정 이상일 경우, 오류로 감지 Fail
    • Skip 내역 Table 관리 -> IMS 화면 조회 -> 후처리 기능 -> Batch 호출

4. 재처리

  • 각 Job별 후처리를 위한 Key 기반의 파라미터 정의

5. Batch 시, 실행결과 조회 및 후처리 기능

6. 테스트 기간을 축소하기 위한 테스트 코드 작성

  • 테스트 지원을 위한 테스트 코드

7. JPA VS mybatis/JDBC Cursor template

기능기술이유
도메인 규칙 처리 소량 조회JPA1차 캐싱, 영속성 컨테스트 관리로 인한 메모리 사용, Flush/clear 비용 등으로 인한 성능 저하 가능성
대량의 데이터 조회(Reader)JDBC Cursor메모리에 한 번에 올리지 않고 스트리밍 형태로 받아옴
대량 Insert/UpdateJDBC+MERGE한 번의 건을 Chunk 트랜잭션 커밋 단위로 처리
